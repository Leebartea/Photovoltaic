# Backend Admin Console Guide

## Purpose

The backend now serves a minimal same-origin admin console at:

- `/admin`

This is a narrow operations surface for premium sync administration.

It is meant for:

- loading backend security posture
- issuing short-lived seat sessions
- inspecting shared seats
- requesting and reviewing high-risk seat-action approvals
- issuing one-time seat invites
- issuing one-time recovery codes
- recording invite / recovery handoff delivery
- preparing provider-ready dispatch packs for active invite / recovery items
- acknowledging when a recorded handoff has actually been closed
- pulling recent backend audit entries

It is **not** a second calculator UI.

It does not run sizing math, and it does not replace the main static app.

## Routes

The backend now exposes:

- `GET /admin`
- `GET /admin/app.js`
- `GET /admin/app.css`
- `GET /api/admin/session-context`
- `GET /api/admin/audit-export`
- `GET /api/admin/action-approvals`
- `POST /api/admin/action-approvals/request`
- `POST /api/admin/action-approvals/review`
- `GET /api/admin/delivery-trail`
- `POST /api/admin/delivery-trail/record`
- `POST /api/admin/delivery-trail/acknowledge`
- `POST /api/admin/delivery-dispatch/prepare`
- `GET /api/admin/posture`

The console then uses the existing protected premium routes behind the scenes:

- `GET /api/team-seats`
- `POST /api/seat-session/issue`
- `POST /api/seat-session/renew`
- `POST /api/seat-session/revoke`
- `POST /api/team-seats/invite/issue`
- `POST /api/team-seats/recovery-code/issue`
- `GET /api/audit-log`
- `GET /api/admin/audit-export`
- `GET /api/admin/delivery-trail`
- `POST /api/admin/delivery-trail/record`
- `POST /api/admin/delivery-trail/acknowledge`
- `POST /api/admin/delivery-dispatch/prepare`
- `POST /api/team-seats/recovery`

The shell-unlock state comes from:

- `GET /api/admin/session-context?installationKey=...`

That route accepts only a valid `X-Session-Token`.

## Security Posture

The admin console is intentionally conservative:

- same-origin only
- no browser local-storage secret persistence
- API key stays memory-only
- seat access code stays memory-only
- session token stays memory-only
- frontend app URL is optional and only used to construct signed handoff links
- admin actions still go through the existing backend permission checks
- posture review now also surfaces premium ops readiness, so missing backup ownership or incident-runbook fields show up before an operator treats the deployment as public-paid ready

The console does **not** introduce a softer auth path.

It simply gives operators a safer place to drive the existing backend controls.

## High-Risk Approval Queue

The admin shell now also carries a small approval queue for the highest-risk shared-seat actions:

- `Rotate Access Code`
- `Disable Shared Sign-In`
- `Suspend Seat`

Those actions are no longer treated like normal direct recovery operations.

They now require:

1. an approval request
2. an approval review
3. the normal recovery execution route

This keeps the queue and the final execution inside the same backend trust boundary.

## Authenticated Shell Behavior

The admin console is now intentionally session-locked.

That means:

- posture review is always visible
- session bootstrap is always visible
- shared-seat actions stay visually and functionally locked
- the shell unlocks only after a valid short-lived seat session exists

This is deliberate.

It keeps the normal operator flow tied to a named seat session instead of letting the API key feel like the real day-to-day credential.

## Audit Snapshot Export

The admin shell now supports server-side audit export in:

- `TXT`
- `CSV`

It also now supports a filtered audit timeline in the shell itself, with:

- category filter
- action filter
- free-text contains filter
- date-grouped timeline rendering

This is different from simply copying what the browser already rendered.

The export is generated by the backend from the live audit store and requires:

- a valid short-lived seat session
- a seat that still holds `audit_read`

That keeps the export trustworthy and aligned with the backend permission model.

## Delivery Trail

The admin shell now also supports a dedicated delivery-trail panel for:

- one-time seat invite handoff
- one-time recovery handoff

Operators can:

- choose the delivery channel
- capture a recipient
- capture an artifact mode
- add an operator note
- record the handoff against the backend-issued delivery reference
- reload the filtered trail from the backend

This keeps the premium admin surface from stopping at issuance only.

It also keeps the handoff record server-side and tied to the admin seat session that recorded it.

## Delivery Acknowledgment

The admin shell now also supports explicit delivery acknowledgment.

That means the trail can distinguish between:

- delivery recorded
- delivery acknowledged

Operators can:

- filter by acknowledgment status
- add an acknowledgment note
- mark a pending delivery entry as acknowledged

This closes the invite / recovery handoff loop more cleanly for premium operations.

## Dispatch Pack

The admin shell now also supports a dispatch-pack panel for active invite and recovery items.

This is the operator-facing copy pack, not the handoff record.

It lets the admin shell prepare:

- subject
- body
- TXT snapshot

using:

- the active one-time invite or recovery item
- the selected delivery channel
- the recipient
- the artifact mode
- the frontend app URL when a full handoff link is wanted

Important security choice:

- the backend validates the currently issued code or signed link against the active hashed issue record
- it does not need to persist raw one-time secrets at rest just to generate the pack

## Recommended Usage Flow

1. Open `/admin` from the backend host.
2. Check the security posture banner first.
3. Enter the `installationKey`.
4. Prefer issuing a short-lived seat session with:
   - `Admin Seat ID`
   - `Seat Access Code`
5. Use the API key only as bootstrap or break-glass recovery.
6. Unlock the shell by confirming the session-backed admin context.
7. Load seats.
8. Select the target seat.
9. If the action is high-risk, request and review it in the approval queue first.
10. Issue a one-time invite or recovery code when needed.
11. If you want a clickable handoff link, add the static frontend app URL first.
12. Prepare the dispatch pack if you want a clean email / chat / SMS handoff message.
13. Record how the invite or recovery item was actually handed off.
14. Mark the delivery entry acknowledged when the desk confirms receipt or the handoff closes.
15. Pull the audit log after sensitive actions.

## Frontend App URL

The admin console can build full signed handoff links for:

- seat invite
- seat recovery

It only does that when you provide the static app URL, for example:

- `https://your-static-host.example.com/pv_calculator_ui.html`

That keeps the backend console separate from the calculator UI while still supporting clean handoff links.

If you leave that field blank:

- raw one-time codes still work
- signed tokens still exist server-side
- the console simply will not try to guess your frontend host

## Production Expectations

Before using `/admin` on a real deployment, set:

- `BACKEND_API_KEY` or `BACKEND_API_KEYS`
- `BACKEND_ALLOWED_ORIGINS`
- `BACKEND_ACTION_LINK_SECRET`

The posture endpoint warns when those are still in a weak or fallback mode.

## Honest Boundary

This admin console is the right next backend surface because it helps with:

- premium trust
- seat administration
- invite delivery
- recovery delivery
- invite / recovery handoff traceability
- provider-ready operator handoff copy
- audit visibility

It does **not** justify moving the calculator engine server-side.

Keep the core PV calculator frontend/offline-first.
